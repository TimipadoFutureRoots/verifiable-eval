# Limitations

## What Verification Covers

- **Verifies execution integrity, not evaluation quality.** verifiable-eval confirms that an evaluation was conducted as described — that the configuration was not changed after commitment, that the trace was not tampered with, and that the reported statistics match the trace. It does not assess whether the evaluation was well-designed, whether the rubrics were appropriate, or whether the results are meaningful.
- **Does not prevent fabrication from scratch.** The commit-then-execute protocol prevents *selective modification* of a real evaluation (changing scores, dropping inconvenient results, altering configuration after the fact). It does not prevent someone from fabricating an entire evaluation — config, trace, and certificate — from whole cloth.
- **Does not verify rubric quality.** The tool checks that the declared rubrics were the ones used. It does not assess whether those rubrics actually measure what they claim to measure.

## Cryptographic Guarantees

- **Tamper-evidence, not tamper-prevention.** The SHA-256 hash chain provides evidence of tampering after the fact. It does not prevent modification — anyone with file access can edit the trace. The chain will break, but the edit itself is not blocked.
- **No hardware attestation.** The tool does not currently run inside a Trusted Execution Environment (TEE). There is no hardware-level guarantee that the evaluation code was not modified during execution.
- **No external timestamping.** Timestamps in the trace are generated by the local system clock. There is no integration with an external time-stamping authority (RFC 3161), so timestamps can be spoofed if the system clock is manipulated.
- **Hash chain is only as strong as SHA-256.** While SHA-256 is considered secure for current purposes, the tool does not support algorithm agility or post-quantum hash functions.

## Statistical Metrics

- **Krippendorff's alpha requires sufficient data.** With only two judges and a small number of scenarios, the alpha estimate has high variance and may not be meaningful. The metric is most informative with 3+ judges and 10+ scenarios per axis.
- **Cohen's kappa uses binned scores.** Continuous scores are binned into 4 categories (quartiles) before computing kappa. This loses precision — two scores of 0.24 and 0.26 are placed in different bins despite being nearly identical.
- **Standard deviation is rounded to 4 decimal places at storage time.** For small sample sizes, this can lose precision. Results should be treated as approximate.
- **Score range is [0, 1].** Scores returned by the LLM judge outside this range are clamped. If a judge consistently returns out-of-range scores, the clamped values may not reflect the judge's intended assessment.

## Family Overlap Detection

- **Known-families map may be incomplete.** The family detection relies on a hardcoded mapping of model name prefixes to provider families (Anthropic, OpenAI, Google, Meta, Mistral, DeepSeek, Alibaba, Cohere). New model providers or unusual naming conventions may not be recognised.
- **Family overlap is a flag, not a disqualification.** Same-family configurations (e.g., Claude as both model-under-test and judge) can produce systematic scoring distortions, but overlap does not automatically invalidate results. The tool declares overlap for transparency; interpretation is left to the reviewer.
- **Provider-level grouping may be too coarse.** All Anthropic models are grouped as one family, but scoring bias between Claude Haiku and Claude Opus may differ from bias between two Opus variants. The tool does not distinguish within-family model distances.

## LLM Judge

- **API failures return empty or zero scores.** If the judge API is unreachable after retries, `generate()` returns an empty string and `score()` returns 0.0. These failure modes are logged but not distinguishable from genuine outputs in the trace.
- **No judge output validation beyond score extraction.** The tool extracts a numeric score from the judge's response but does not validate that the judge followed the rubric or provided coherent reasoning.
- **Single evaluation run per certificate.** Each certificate corresponds to one execution. There is no mechanism for aggregating results across multiple runs or tracking score stability over time.

## Operational

- **Does not replace domain expertise.** Choosing what to evaluate — which axes, scenarios, rubrics, and judge models to use — requires human judgement and domain knowledge. The tool automates execution and verification, not evaluation design.
- **No incremental or streaming certificates.** The certificate is generated after the full evaluation completes. Long-running evaluations cannot produce intermediate verifiable checkpoints.
- **File-based storage only.** Configuration, traces, and certificates are stored as local JSON/JSONL files. There is no built-in integration with databases, cloud storage, or certificate registries.
